# Исправление системы модерации регистраций

## Проблема
Система автоматически одобряет регистрацию пользователей вместо отправки на модерацию.

## Причина
В базе данных уже есть пользователи со статусом `approved` и `isactive = true`, созданные до внедрения системы модерации.

## Решение

### Шаг 1: Выполните SQL скрипт для сброса статусов

Выполните скрипт `reset_user_statuses.sql` в вашей базе данных:

```sql
-- Обновляем всех пользователей кроме администраторов
UPDATE users 
SET 
    registrationstatus = 'pending',
    isactive = false,
    rejectionreason = NULL,
    moderatedat = NULL,
    moderatedby = NULL
WHERE role != 'Admin';
```

**ВАЖНО:** Этот скрипт переведёт всех существующих пользователей (кроме админов) в статус ожидания модерации. Если у вас есть активные пользователи, которых не нужно переводить в pending, измените условие WHERE.

### Шаг 2: Перезапустите приложение

```bash
dotnet run
```

### Шаг 3: Проверьте работу системы

1. **Зарегистрируйте нового пользователя** через форму регистрации
2. **Проверьте в базе данных:**
   ```sql
   SELECT id, email, fullname, registrationstatus, isactive 
   FROM users 
   ORDER BY createdat DESC 
   LIMIT 5;
   ```
   Новый пользователь должен иметь `registrationstatus = 'pending'` и `isactive = false`

3. **Войдите как администратор** и перейдите в раздел "Модерация регистраций"
4. **Проверьте список заявок** - там должны быть пользователи со статусом "pending"
5. **Одобрите или отклоните заявку**

### Шаг 4: Проверьте что новые пользователи не могут войти до одобрения

1. Попробуйте войти под новым зарегистрированным пользователем
2. Система должна показать ошибку "Ваша учетная запись неактивна"
3. После одобрения администратором пользователь сможет войти

## Проверка полей в базе данных

Убедитесь что в таблице `users` есть все необходимые поля:

```sql
SELECT column_name, data_type, is_nullable
FROM information_schema.columns 
WHERE table_name = 'users' 
AND column_name IN ('registrationstatus', 'rejectionreason', 'moderatedat', 'moderatedby')
ORDER BY column_name;
```

Должны быть:
- `registrationstatus` (varchar/text)
- `rejectionreason` (varchar/text, nullable)
- `moderatedat` (timestamp, nullable)
- `moderatedby` (integer, nullable)

## Альтернативный вариант (без сброса статусов)

Если вы НЕ хотите переводить существующих пользователей в pending, просто:

1. Убедитесь что скрипт `add_registration_status.sql` выполнен
2. Перезапустите приложение
3. Новые регистрации будут требовать модерации
4. Старые пользователи останутся активными

## Что делает система модерации

### При регистрации (AuthController.cs):
- Создаёт пользователя со статусом `pending`
- Устанавливает `isactive = false`
- НЕ выдаёт JWT токен
- Возвращает сообщение "Ваша заявка отправлена на модерацию"

### При входе (AuthController.cs):
- Проверяет `isactive`
- Если `false` - возвращает ошибку "Ваша учетная запись неактивна"

### При одобрении (UsersController.cs):
- Устанавливает `registrationstatus = 'approved'`
- Устанавливает `isactive = true`
- Записывает дату и ID модератора
- Отправляет уведомление пользователю

### При отклонении (UsersController.cs):
- Устанавливает `registrationstatus = 'rejected'`
- Сохраняет причину отклонения
- Отправляет уведомление пользователю

## Проверка логов

Если проблема сохраняется, проверьте:

1. **Консоль браузера (F12)** - есть ли ошибки JavaScript
2. **Консоль приложения** - есть ли ошибки при запросах к API
3. **Сетевая вкладка (Network)** - какие данные возвращает API

## Контакты API

- `POST /api/auth/register` - регистрация (создаёт pending пользователя)
- `GET /api/users/pending-registrations` - список заявок (только для админа)
- `POST /api/users/{id}/approve` - одобрить заявку (только для админа)
- `POST /api/users/{id}/reject` - отклонить заявку (только для админа)
