# Требования к программе лояльности клиентов

## Обзор

Программа лояльности предназначена для поощрения постоянных клиентов магазина инструментов путем предоставления скидок на основе общей суммы их заказов. Система автоматически отслеживает накопленную сумму покупок каждого клиента и позволяет администраторам назначать процент скидки при достижении определенных порогов.

## Требование 1

**User Story:** Как клиент, я хочу видеть свою текущую скидку и прогресс до следующего уровня, чтобы понимать выгоду от покупок в магазине.

### Acceptance Criteria

1. WHEN клиент открывает свой профиль THEN система SHALL отображать текущий процент скидки
2. WHEN клиент открывает свой профиль THEN система SHALL отображать общую сумму всех заказов
3. WHEN клиент открывает свой профиль THEN система SHALL отображать прогресс до следующего порога скидки
4. WHEN клиент не имеет скидки THEN система SHALL отображать "0%" и сумму до первого порога
5. WHEN клиент достиг максимального порога THEN система SHALL отображать максимальную скидку без прогресс-бара

## Требование 2

**User Story:** Как система, я должна автоматически накапливать общую сумму заказов клиента, чтобы отслеживать его покупательскую активность.

### Acceptance Criteria

1. WHEN создается новый пользователь THEN система SHALL инициализировать поле TotalOrdersAmount значением 0
2. WHEN заказ переходит в статус "Completed" THEN система SHALL добавить сумму заказа к TotalOrdersAmount пользователя
3. WHEN заказ отменяется THEN система SHALL вычесть сумму заказа из TotalOrdersAmount пользователя
4. WHEN происходит изменение TotalOrdersAmount THEN система SHALL сохранить изменения в базе данных немедленно
5. WHEN запрашивается TotalOrdersAmount THEN система SHALL возвращать актуальное значение из базы данных

## Требование 3

**User Story:** Как администратор, я хочу назначать процент скидки клиентам при достижении порогов, чтобы поощрять их лояльность.

### Acceptance Criteria

1. WHEN администратор открывает список пользователей THEN система SHALL отображать TotalOrdersAmount и CurrentDiscount для каждого клиента
2. WHEN администратор выбирает клиента THEN система SHALL предоставить возможность изменить процент скидки
3. WHEN администратор вводит новый процент скидки THEN система SHALL валидировать значение (0-100)
4. WHEN администратор сохраняет новый процент скидки THEN система SHALL обновить поле CurrentDiscount в базе данных
5. WHEN администратор сохраняет новый процент скидки THEN система SHALL отправить уведомление клиенту о изменении скидки

## Требование 4

**User Story:** Как администратор, я хочу видеть рекомендуемые пороги скидок, чтобы принимать обоснованные решения о назначении скидок.

### Acceptance Criteria

1. WHEN администратор просматривает клиента с TotalOrdersAmount >= 1000 BYN THEN система SHALL рекомендовать скидку 3%
2. WHEN администратор просматривает клиента с TotalOrdersAmount >= 5000 BYN THEN система SHALL рекомендовать скидку 5%
3. WHEN администратор просматривает клиента с TotalOrdersAmount >= 10000 BYN THEN система SHALL рекомендовать скидку 10%
4. WHEN администратор просматривает клиента с TotalOrdersAmount < 1000 BYN THEN система SHALL не отображать рекомендации
5. WHEN текущая скидка клиента ниже рекомендуемой THEN система SHALL выделить рекомендацию визуально

## Требование 5

**User Story:** Как система, я должна автоматически применять скидку при оформлении заказа, чтобы клиент получал свою выгоду.

### Acceptance Criteria

1. WHEN клиент добавляет товар в корзину THEN система SHALL отображать цену с учетом его текущей скидки
2. WHEN клиент оформляет заказ THEN система SHALL применить скидку к общей сумме заказа
3. WHEN скидка применяется THEN система SHALL сохранить примененный процент скидки в записи заказа
4. WHEN скидка применяется THEN система SHALL сохранить сумму скидки в записи заказа
5. WHEN клиент не имеет скидки THEN система SHALL использовать полную цену товаров

## Требование 6

**User Story:** Как администратор, я хочу видеть историю изменений скидок клиентов, чтобы отслеживать работу программы лояльности.

### Acceptance Criteria

1. WHEN администратор изменяет скидку клиента THEN система SHALL записать изменение в лог с датой и временем
2. WHEN администратор изменяет скидку клиента THEN система SHALL записать старое и новое значение скидки
3. WHEN администратор изменяет скидку клиента THEN система SHALL записать ID администратора, выполнившего изменение
4. WHEN администратор просматривает клиента THEN система SHALL отображать последние 10 изменений скидки
5. WHEN запрашивается полная история THEN система SHALL предоставить все записи изменений для клиента

## Требование 7

**User Story:** Как клиент, я хочу получать уведомления о изменении моей скидки, чтобы быть в курсе своих привилегий.

### Acceptance Criteria

1. WHEN администратор увеличивает скидку клиента THEN система SHALL создать уведомление для клиента
2. WHEN администратор уменьшает скидку клиента THEN система SHALL создать уведомление для клиента
3. WHEN создается уведомление о скидке THEN оно SHALL содержать старый и новый процент скидки
4. WHEN создается уведомление о скидке THEN оно SHALL содержать дату изменения
5. WHEN клиент открывает уведомления THEN система SHALL отображать уведомления о скидках с соответствующей иконкой

## Требование 8

**User Story:** Как система, я должна корректно обрабатывать граничные случаи, чтобы обеспечить надежность программы лояльности.

### Acceptance Criteria

1. WHEN TotalOrdersAmount становится отрицательным из-за отмен THEN система SHALL установить значение в 0
2. WHEN администратор вводит процент скидки > 100 THEN система SHALL отклонить ввод с сообщением об ошибке
3. WHEN администратор вводит отрицательный процент скидки THEN система SHALL отклонить ввод с сообщением об ошибке
4. WHEN клиент имеет скидку и заказ отменяется THEN система SHALL пересчитать TotalOrdersAmount но сохранить текущую скидку
5. WHEN происходит одновременное изменение скидки двумя администраторами THEN система SHALL применить последнее изменение и записать оба в лог

## Требование 9

**User Story:** Как разработчик, я хочу иметь API для управления скидками, чтобы интегрировать функциональность в существующую систему.

### Acceptance Criteria

1. WHEN вызывается GET /api/users/{id}/loyalty THEN система SHALL возвращать TotalOrdersAmount и CurrentDiscount
2. WHEN вызывается PUT /api/users/{id}/discount THEN система SHALL обновить CurrentDiscount с валидацией
3. WHEN вызывается GET /api/users/{id}/discount-history THEN система SHALL возвращать историю изменений скидок
4. WHEN вызывается POST /api/orders/{id}/complete THEN система SHALL обновить TotalOrdersAmount пользователя
5. WHEN API возвращает ошибку THEN она SHALL содержать понятное сообщение и соответствующий HTTP статус код

## Требование 10

**User Story:** Как администратор, я хочу фильтровать и сортировать клиентов по параметрам лояльности, чтобы эффективно управлять программой.

### Acceptance Criteria

1. WHEN администратор открывает список клиентов THEN система SHALL предоставить фильтр по диапазону TotalOrdersAmount
2. WHEN администратор открывает список клиентов THEN система SHALL предоставить фильтр по диапазону CurrentDiscount
3. WHEN администратор применяет фильтры THEN система SHALL отображать только соответствующих клиентов
4. WHEN администратор сортирует по TotalOrdersAmount THEN система SHALL упорядочить клиентов по возрастанию/убыванию
5. WHEN администратор сортирует по CurrentDiscount THEN система SHALL упорядочить клиентов по возрастанию/убыванию

## Нефункциональные требования

### Производительность

1. Обновление TotalOrdersAmount должно выполняться за < 100ms
2. Загрузка страницы профиля клиента с данными лояльности должна выполняться за < 500ms
3. API запросы должны обрабатываться за < 200ms

### Безопасность

1. Только администраторы могут изменять скидки клиентов
2. Клиенты могут видеть только свои собственные данные лояльности
3. Все изменения скидок должны логироваться с указанием автора

### Надежность

1. Изменения TotalOrdersAmount должны быть транзакционными
2. При сбое обновления скидки система должна откатить изменения
3. История изменений скидок должна храниться постоянно

### Совместимость

1. Функциональность должна интегрироваться с существующей системой заказов
2. Должна поддерживаться работа с PostgreSQL базой данных
3. Frontend должен быть адаптирован для мобильных устройств
