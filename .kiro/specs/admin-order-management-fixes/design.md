# Исправление багов управления заказами - Технический дизайн

## Обзор

Данный документ описывает технический подход к исправлению двух критических багов в функционале управления заказами:

1. **Баг оформления заказа**: Кнопка "Оформить заказ" в корзине не работает из-за отсутствия поддержки динамических модальных окон в функции `showModal`
2. **Баг просмотра деталей администратором**: Отсутствует функционал отображения полной информации о клиенте (физ./юр. лицо) для администраторов

Оба бага критически влияют на основной бизнес-процесс и требуют немедленного исправления.

## Глоссарий

- **Bug_Condition (C)**: Условие, при котором проявляется баг
- **Property (P)**: Ожидаемое корректное поведение при выполнении условия C
- **Preservation**: Существующее поведение, которое должно остаться неизменным после исправления
- **showModal**: Функция в `wwwroot/js/modules/auth.js`, которая отображает модальное окно по его ID
- **checkout**: Функция в `wwwroot/js/modules/cart.js`, которая инициирует процесс оформления заказа
- **viewOrderDetails**: Функция в `wwwroot/js/modules/orders.js`, которая отображает детали заказа
- **currentUser**: Объект в `app.js`, содержащий информацию о текущем пользователе, включая роль (role)
- **isLegalEntity**: Флаг в данных заказа, указывающий, является ли клиент юридическим лицом

## Детали багов

### Баг 1: Условие проявления бага оформления заказа

Баг проявляется когда пользователь пытается оформить заказ из корзины. Функция `checkout()` в `wwwroot/js/modules/cart.js` вызывает `showCheckoutModal()`, которая создает HTML-контент модального окна и пытается вызвать `this.showModal('checkout-modal', modalContent)` с двумя параметрами. Однако функция `showModal` в `wwwroot/js/modules/auth.js` принимает только один параметр `modalId` и ожидает, что элемент с таким ID уже существует в DOM.

**Формальная спецификация:**
```
FUNCTION isBugCondition_Checkout(input)
  INPUT: input of type UserAction
  OUTPUT: boolean
  
  RETURN input.action === 'clickCheckoutButton'
         AND cart.totalItems > 0
         AND modalElement('checkout-modal') NOT EXISTS in DOM
         AND showModal called with (modalId, modalContent)
         AND showModal expects only modalId parameter
END FUNCTION
```

### Баг 2: Условие проявления бага просмотра деталей администратором

Баг проявляется когда администратор (role === 'Admin' OR role === 'Manager') пытается просмотреть детали заказа. Функция `viewOrderDetails()` в `wwwroot/js/modules/orders.js` создает модальное окно с информацией о заказе, но:
1. Использует тот же паттерн вызова `showModal` с двумя параметрами
2. Отображает информацию о клиенте только для администраторов, но эта логика уже реализована корректно

**Формальная спецификация:**
```
FUNCTION isBugCondition_AdminOrderDetails(input)
  INPUT: input of type UserAction
  OUTPUT: boolean
  
  RETURN input.action === 'clickOrderDetailsButton'
         AND (currentUser.role === 'Admin' OR currentUser.role === 'Manager')
         AND modalElement('order-details-modal') NOT EXISTS in DOM
         AND showModal called with (modalId, modalContent)
         AND showModal expects only modalId parameter
END FUNCTION
```

### Примеры проявления

**Баг 1 - Оформление заказа:**
- Пользователь добавляет товары в корзину
- Нажимает кнопку "Оформить заказ"
- **Ожидается**: Открывается модальное окно с формой комментария
- **Фактически**: Ничего не происходит, модальное окно не открывается

**Баг 2 - Просмотр деталей администратором:**
- Администратор открывает страницу "Заказы"
- Нажимает кнопку "Детали" около заказа
- **Ожидается**: Открывается модальное окно с полной информацией о заказе и клиенте
- **Фактически**: Ничего не происходит, модальное окно не открывается

**Граничный случай:**
- Обычный пользователь (role === 'Customer') просматривает свои заказы
- **Ожидается**: Видит кнопку "Детали" и может просмотреть состав заказа БЕЗ регистрационных данных
- **Должно работать**: После исправления обычные пользователи не должны видеть регистрационные данные

## Ожидаемое поведение

### Требования к сохранению существующего поведения

**Неизменное поведение:**
- Добавление товаров в корзину должно продолжать работать корректно
- Изменение количества товара в корзине должно корректно пересчитывать итоговую сумму
- Удаление товара из корзины должно корректно обновлять корзину
- Обычные пользователи должны видеть кнопку "Детали" для просмотра состава заказа без регистрационных данных
- Администраторы должны видеть список товаров, статус заказа, дату создания и общую сумму
- Сохранение комментария к заказу должно работать корректно
- Создание заказа без комментария должно работать корректно
- Фильтрация заказов по статусу должна работать корректно

**Область действия:**
Все функции корзины и заказов, не связанные с открытием модальных окон, должны остаться полностью неизменными. Это включает:
- Управление товарами в корзине (добавление, удаление, изменение количества)
- Отображение списка заказов
- Фильтрацию заказов
- Отображение информации о заказе (после открытия модального окна)

## Гипотеза о первопричине

На основе анализа кода, наиболее вероятные причины:

1. **Несоответствие сигнатуры функции showModal**: Функция `showModal` в `wwwroot/js/modules/auth.js` (строки 7-12) принимает только один параметр `modalId` и ожидает, что элемент с таким ID уже существует в DOM. Однако код в `cart.js` (строка 211) и `orders.js` (строка 311) вызывает её с двумя параметрами: `showModal(modalId, modalContent)`.

2. **Отсутствие статических модальных окон в HTML**: В файле `wwwroot/index.html` отсутствуют предопределенные элементы `<div id="checkout-modal">` и `<div id="order-details-modal">`. Все существующие модальные окна (product-modal, reviews-modal, faq-modal и т.д.) имеют статическую структуру в HTML.

3. **Отсутствие функции для динамического создания модальных окон**: В кодовой базе нет функции, которая могла бы создавать модальные окна динамически на основе переданного HTML-контента.

## Корректные свойства

Property 1: Bug Condition - Оформление заказа через модальное окно

_For any_ пользовательское действие, при котором пользователь нажимает кнопку "Оформить заказ" в корзине с товарами, исправленная функция checkout SHALL открыть модальное окно с формой для ввода комментария, позволить пользователю подтвердить заказ, создать заказ на сервере и перенаправить пользователя на страницу "Заказы" с уведомлением об успешном оформлении.

**Validates: Requirements 2.1, 2.2, 2.3, 2.4**

Property 2: Bug Condition - Просмотр деталей заказа администратором

_For any_ пользовательское действие, при котором администратор (role === 'Admin' OR role === 'Manager') нажимает кнопку "Детали заказа", исправленная функция viewOrderDetails SHALL открыть модальное окно с полной информацией о заказе, включая регистрационные данные клиента (ФИО и Email для физ. лиц; companyname, unp, contactperson, contactphone, legaladdress, actualaddress, directorname, bankname, bankcode, checkingaccount для юр. лиц), но НЕ отображать пароль пользователя.

**Validates: Requirements 2.5, 2.6, 2.7, 2.8, 2.9**

Property 3: Preservation - Функционал корзины и заказов

_For any_ пользовательское действие, которое НЕ связано с открытием модальных окон оформления заказа или просмотра деталей (добавление/удаление товаров, изменение количества, фильтрация заказов), исправленный код SHALL производить точно такой же результат, как и оригинальный код, сохраняя все существующие функции корзины и управления заказами.

**Validates: Requirements 3.1, 3.2, 3.3, 3.4, 3.5, 3.6, 3.7, 3.8**

Property 4: Preservation - Разграничение доступа

_For any_ обычного пользователя (role === 'Customer'), который просматривает свои заказы, исправленный код SHALL НЕ отображать регистрационные данные в деталях заказа, сохраняя текущее поведение отображения только состава заказа, статуса и суммы.

**Validates: Requirements 2.10, 3.4**

## Реализация исправления

### Необходимые изменения

Исходя из корректности анализа первопричины:

**Файл**: `wwwroot/js/modules/auth.js`

**Функция**: `showModal`

**Конкретные изменения**:

1. **Расширение функции showModal для поддержки динамического контента**:
   - Добавить второй необязательный параметр `content` в функцию `showModal(modalId, content = null)`
   - Если `content` передан, создать динамическое модальное окно с этим контентом
   - Если `content` не передан, использовать существующую логику для статических модальных окон

2. **Создание динамического модального окна**:
   - Проверить, существует ли элемент с `modalId` в DOM
   - Если не существует и передан `content`, создать новый элемент `<div class="modal-overlay" id="${modalId}">`
   - Вставить `content` внутрь модального окна
   - Добавить элемент в `document.body`

3. **Обработка закрытия динамических модальных окон**:
   - При закрытии динамического модального окна удалить его из DOM
   - Модифицировать функцию `closeModal` для удаления динамических модальных окон

4. **Обработка клика вне модального окна**:
   - Добавить обработчик события клика на `modal-overlay` для закрытия модального окна при клике вне его области

**Файл**: `wwwroot/js/modules/cart.js`

**Функция**: `showCheckoutModal`

**Конкретные изменения**:
- Код уже корректен, изменения не требуются
- После исправления `showModal` функция будет работать корректно

**Файл**: `wwwroot/js/modules/orders.js`

**Функция**: `viewOrderDetails`

**Конкретные изменения**:
- Код уже корректен, изменения не требуются
- Логика отображения информации о клиенте для администраторов уже реализована
- После исправления `showModal` функция будет работать корректно

## Стратегия тестирования

### Подход к валидации

Стратегия тестирования следует двухфазному подходу: сначала выявить контрпримеры, демонстрирующие баг на неисправленном коде, затем проверить, что исправление работает корректно и сохраняет существующее поведение.

### Исследовательская проверка условия бага

**Цель**: Выявить контрпримеры, демонстрирующие баг ДО реализации исправления. Подтвердить или опровергнуть анализ первопричины. Если опровергнем, потребуется пересмотр гипотезы.

**План тестирования**: Написать тесты, которые симулируют действия пользователя (нажатие кнопки "Оформить заказ" и "Детали заказа") и проверяют, что модальные окна открываются. Запустить эти тесты на НЕИСПРАВЛЕННОМ коде для наблюдения за ошибками и понимания первопричины.

**Тестовые случаи**:
1. **Тест оформления заказа**: Добавить товары в корзину, нажать "Оформить заказ", проверить, что модальное окно открылось (провалится на неисправленном коде)
2. **Тест просмотра деталей администратором**: Войти как администратор, открыть список заказов, нажать "Детали", проверить, что модальное окно открылось (провалится на неисправленном коде)
3. **Тест просмотра деталей обычным пользователем**: Войти как обычный пользователь, открыть список заказов, нажать "Детали", проверить, что модальное окно открылось без регистрационных данных (провалится на неисправленном коде)
4. **Тест граничного случая**: Попытаться оформить заказ с пустой корзиной (должен показать ошибку, а не открывать модальное окно)

**Ожидаемые контрпримеры**:
- Модальные окна не открываются при нажатии кнопок
- Возможные причины: отсутствие элемента в DOM, несоответствие сигнатуры функции, отсутствие обработки динамического контента

### Проверка исправления

**Цель**: Проверить, что для всех входных данных, где выполняется условие бага, исправленная функция производит ожидаемое поведение.

**Псевдокод:**
```
FOR ALL input WHERE isBugCondition_Checkout(input) DO
  result := checkout_fixed(input)
  ASSERT modalOpened(result) AND modalContainsCommentForm(result)
END FOR

FOR ALL input WHERE isBugCondition_AdminOrderDetails(input) DO
  result := viewOrderDetails_fixed(input)
  ASSERT modalOpened(result) AND modalContainsClientInfo(result)
END FOR
```

### Проверка сохранения поведения

**Цель**: Проверить, что для всех входных данных, где НЕ выполняется условие бага, исправленная функция производит тот же результат, что и оригинальная функция.

**Псевдокод:**
```
FOR ALL input WHERE NOT isBugCondition_Checkout(input) AND NOT isBugCondition_AdminOrderDetails(input) DO
  ASSERT cartFunctions_original(input) = cartFunctions_fixed(input)
  ASSERT ordersFunctions_original(input) = ordersFunctions_fixed(input)
END FOR
```

**Подход к тестированию**: Property-based тестирование рекомендуется для проверки сохранения поведения, потому что:
- Оно автоматически генерирует множество тестовых случаев по всему домену входных данных
- Оно выявляет граничные случаи, которые могут быть пропущены в ручных unit-тестах
- Оно предоставляет сильные гарантии, что поведение не изменилось для всех входных данных, не связанных с багом

**План тестирования**: Наблюдать поведение на НЕИСПРАВЛЕННОМ коде для действий с корзиной и заказами (добавление товаров, изменение количества, удаление, фильтрация), затем написать property-based тесты, фиксирующие это поведение.

**Тестовые случаи**:
1. **Сохранение функций корзины**: Проверить, что добавление, удаление и изменение количества товаров работает так же, как до исправления
2. **Сохранение отображения заказов**: Проверить, что список заказов отображается корректно после исправления
3. **Сохранение фильтрации заказов**: Проверить, что фильтрация по статусу работает корректно
4. **Сохранение разграничения доступа**: Проверить, что обычные пользователи не видят регистрационные данные

### Unit-тесты

- Тест функции `showModal` с одним параметром (статическое модальное окно)
- Тест функции `showModal` с двумя параметрами (динамическое модальное окно)
- Тест функции `closeModal` для статических модальных окон
- Тест функции `closeModal` для динамических модальных окон (должно удалить из DOM)
- Тест функции `checkout` с непустой корзиной
- Тест функции `checkout` с пустой корзиной (граничный случай)
- Тест функции `viewOrderDetails` для администратора
- Тест функции `viewOrderDetails` для обычного пользователя

### Property-Based тесты

- Генерировать случайные состояния корзины и проверять, что функции корзины работают одинаково до и после исправления
- Генерировать случайные списки заказов и проверять, что фильтрация работает одинаково
- Генерировать случайные роли пользователей и проверять, что разграничение доступа работает корректно

### Интеграционные тесты

- Полный флоу оформления заказа: добавить товары в корзину → нажать "Оформить заказ" → ввести комментарий → подтвердить → проверить перенаправление на страницу "Заказы"
- Полный флоу просмотра деталей администратором: войти как администратор → открыть список заказов → нажать "Детали" → проверить отображение регистрационных данных
- Полный флоу просмотра деталей обычным пользователем: войти как пользователь → открыть список заказов → нажать "Детали" → проверить отсутствие регистрационных данных
- Тест закрытия модального окна при клике вне его области
- Тест закрытия модального окна при нажатии кнопки "Закрыть"
